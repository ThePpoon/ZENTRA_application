# 7.2 เทคนิคหรือเทคโนโลยีที่ใช้

ระบบ ZENTRA ประยุกต์ใช้เทคโนโลยีปัญญาประดิษฐ์และการประมวลผลภาพในเชิงลึกร่วมกัน เพื่อวิเคราะห์ความปลอดภัยในโรงงานอุตสาหกรรมแบบ Real-time โดยสามารถแบ่งเทคนิคและเทคโนโลยีที่ใช้ออกเป็น 3 ส่วนหลัก ได้แก่ ปัญญาประดิษฐ์และการประมวลผลภาพ อัลกอริธึม และสถาปัตยกรรมซอฟต์แวร์

## 7.2.1 ปัญญาประดิษฐ์และการประมวลผลภาพ (AI & Computer Vision)

ระบบ ZENTRA ใช้โมเดลปัญญาประดิษฐ์หลายชั้นที่ทำงานร่วมกันบนกล้องชุดเดียว โดยแต่ละโมเดลมีหน้าที่เฉพาะทางและส่งต่อข้อมูลให้กันในรูปแบบ Pipeline

### 1) YOLOv8m (You Only Look Once Version 8, Medium)

YOLOv8m คือโมเดลตรวจจับวัตถุ (Object Detection) ที่พัฒนาโดย Ultralytics ใช้หลักการ Single-pass Inference บน Convolutional Neural Network (CNN) กล่าวคือโมเดลจะสแกนภาพเพียงครั้งเดียวในการตรวจจับวัตถุทั้งหมดในภาพ แตกต่างจากสถาปัตยกรรมรุ่นก่อนที่ต้องสแกนภาพหลายรอบ ทำให้มีความเร็วในการประมวลผลสูงและเหมาะสำหรับงาน Real-time

ในระบบ ZENTRA YOLOv8m ทำหน้าที่ตรวจจับอุปกรณ์ป้องกันภัยส่วนบุคคล (PPE) ดังนี้

1. **การตรวจจับ PPE 5 ประเภทหลัก**: หมวกนิรภัย (Helmet), เสื้อกั๊กสะท้อนแสง (Safety Vest), แว่นตานิรภัย (Safety Goggles), ถุงมือ (Gloves) และรองเท้าบูทนิรภัย (Safety Boots)

2. **โครงสร้าง Class**: โมเดลถูกฝึกให้รู้จัก 18 class ซึ่งครอบคลุมทั้งสถานะ "สวม" และ "ไม่สวม" PPE เช่น helmet / no_helmet, vest / no_vest เป็นต้น รวมถึง class "person" สำหรับใช้ร่วมกับโมดูล Safety Zone

3. **พารามิเตอร์การ Inference**:
   - Input Resolution: 640 × 640 pixels
   - Confidence Threshold: 0.45 (กรองเฉพาะ detection ที่มั่นใจ ≥ 45%)
   - IoU Threshold สำหรับ Non-Maximum Suppression (NMS): 0.45

4. **การ Fine-tune โมเดล**: โมเดลพื้นฐาน YOLOv8m.pt ถูก Fine-tune ด้วย Dataset จาก Roboflow PPE Workspace บน GPU โดยใช้พารามิเตอร์การฝึกดังนี้
   - จำนวน Epoch: 100
   - Batch Size: 16
   - Learning Rate เริ่มต้น (lr0): 0.001, สิ้นสุด (lrf): 0.01
   - Momentum: 0.937
   - Weight Decay: 0.0005
   - Warmup Epochs: 5
   - Image Size: 640 × 640
   - Validation Split: 15%
   - เป้าหมายความแม่นยำ: mAP ≥ 85%

5. **การ Deployment**: โมเดลให้บริการผ่าน Roboflow Inference Server ที่รันบน Docker Container บน Local Network (http://localhost:9001) ทำให้ข้อมูลภาพวิดีโอไม่ส่งออกนอกโรงงาน สอดคล้องกับพระราชบัญญัติคุ้มครองข้อมูลส่วนบุคคล (PDPA)

### 2) MediaPipe Pose (BlazePose GHUM)

MediaPipe Pose คือโมเดล Pose Estimation ที่พัฒนาโดย Google ใช้สถาปัตยกรรม BlazePose GHUM ที่สามารถตรวจจับ **33 จุดสำคัญของร่างกาย (Keypoints)** พร้อมค่าความมั่นใจ (visibility confidence) ในแต่ละจุดแบบ Real-time

ในระบบ ZENTRA MediaPipe Pose ทำหน้าที่วิเคราะห์ท่าทางของบุคคลใน **โมดูล Heat Stroke Risk Detection** โดยมีการตั้งค่าดังนี้

1. **พารามิเตอร์โมเดล**:
   - Model Complexity: 1 (ระดับกลาง สมดุลระหว่างความแม่นยำและความเร็ว)
   - Min Detection Confidence: 0.5
   - Min Tracking Confidence: 0.5
   - Smooth Landmarks: เปิดใช้งาน (ลด Noise จาก temporal jitter)
   - Enable Segmentation: ปิดใช้งาน (ไม่จำเป็นสำหรับงานนี้ ประหยัดทรัพยากร)

2. **Keypoints ที่ใช้ในการวิเคราะห์การล้ม**:

   | Index | ชื่อ Keypoint | บทบาทในระบบ |
   |-------|--------------|-------------|
   | 0 | NOSE | ตำแหน่งศีรษะ |
   | 11 | LEFT_SHOULDER | วัดสัดส่วนกรอบร่างกาย |
   | 12 | RIGHT_SHOULDER | วัดสัดส่วนกรอบร่างกาย |
   | 23 | LEFT_HIP | คำนวณจุดศูนย์ถ่วง (COG) |
   | 24 | RIGHT_HIP | คำนวณจุดศูนย์ถ่วง (COG) |
   | 27 | LEFT_ANKLE | ตำแหน่งเท้า |
   | 28 | RIGHT_ANKLE | ตำแหน่งเท้า |

3. **กระบวนการประมวลผล**: ทุก Frame ภาพจะถูกแปลงจาก BGR เป็น RGB (ตามที่ MediaPipe ต้องการ) ก่อนส่งเข้าโมเดล ผลลัพธ์คือ array ขนาด (33, 3) ที่เก็บค่า x, y (normalized 0–1 เทียบกับขนาด frame) และ visibility ของแต่ละ keypoint

4. **Fallback Mechanism**: หากไม่สามารถติดตั้ง MediaPipe ได้ ระบบจะสลับไปใช้ Roboflow Fall Detection Model โดยอัตโนมัติ (Graceful Degradation)

### 3) OpenCV (Open Source Computer Vision Library)

OpenCV เวอร์ชัน ≥ 4.9.0 ทำหน้าที่เป็น Foundation Layer ของระบบ ครอบคลุมทุกขั้นตอนของการประมวลผลภาพ ได้แก่

1. **การรับภาพจากกล้อง**: รองรับ 3 แหล่งข้อมูล — Webcam (DirectShow backend บน Windows 11 ให้ความเร็วสูงกว่า FFMPEG), RTSP Stream จาก IP Camera และไฟล์วิดีโอ
2. **การ Annotate ภาพ**: วาดกรอบ Bounding Box, Polygon Zone, Track Trail, และข้อความ OSD (On-Screen Display) บน Frame แบบ Real-time
3. **การแปลงสีและ Encode**: แปลง BGR ↔ RGB สำหรับ MediaPipe, Encode JPEG สำหรับ Upload แจ้งเตือน
4. **การตรวจสอบ Point-in-Polygon**: ใช้ cv2.pointPolygonTest() สำหรับโมดูล Safety Zone

### 4) Roboflow Inference Server

Roboflow Inference Server ทำหน้าที่เป็น **Local AI Model Serving Platform** รันผ่าน Docker Container ช่วยให้สามารถเรียกใช้โมเดล YOLOv8 ผ่าน HTTP REST API ภายใน Local Network โดยไม่ต้องส่งภาพออกภายนอก

1. **Docker Image**: roboflow/roboflow-inference-server-cpu:latest
2. **Endpoint**: http://localhost:9001
3. **รองรับหลายโมเดลพร้อมกัน**: PPE Detection Model และ Fall Detection Model สามารถให้บริการได้ในเวลาเดียวกันบน Server ชุดเดียว
4. **Inference Frequency**: ระบบส่ง Frame ไป Inference ทุก 3 Frame (INFER_EVERY_N_FRAMES = 3) เพื่อลดภาระประมวลผลและเพิ่มความราบรื่นของการแสดงผล

## 7.2.2 อัลกอริธึม (Algorithms)

### 1) ByteTrack Multi-Object Tracking Algorithm

ByteTrack เป็นอัลกอริธึมติดตามวัตถุหลายชิ้นพร้อมกัน (Multi-Object Tracking: MOT) ที่แก้ปัญหา ID-Switch ซึ่งเกิดขึ้นเมื่อบุคคลถูกบดบัง (Occlusion) โดยความแตกต่างจากอัลกอริธึมทั่วไปคือ ByteTrack **ใช้ทั้ง detection ที่มี confidence สูงและต่ำ** แทนที่จะตัดทิ้ง detection ที่มี confidence ต่ำออกไป

ระบบ ZENTRA พัฒนา ByteTrack ขึ้นเองด้วย NumPy โดยไม่พึ่งพา Library ภายนอก มีขั้นตอนการทำงานดังนี้

**ขั้นตอนที่ 1 — แยกประเภท Detection**
- High-score detections: confidence ≥ 0.50
- Low-score detections: confidence ระหว่าง 0.10–0.50

**ขั้นตอนที่ 2 — จับคู่ High-score กับ Tracked Tracks (IoU Matrix)**

คำนวณเมทริกซ์ IoU (Intersection over Union) ระหว่าง Tracked tracks ทุกตัวกับ High-score detections ทุกตัว

$$\text{IoU} = \frac{\text{พื้นที่ Intersection}}{\text{พื้นที่ Union}} = \frac{\max(0,\ x2_{min} - x1_{max}) \times \max(0,\ y2_{min} - y1_{max})}{A_{track} + A_{det} - \text{Intersection}}$$

จากนั้นใช้ Greedy Matching โดยเรียงคู่จาก IoU สูงสุดก่อน และจับคู่โดยกำหนดว่าแต่ละ track และแต่ละ detection จะถูกจับคู่ได้เพียงครั้งเดียว (IoU Matching Threshold = 0.80)

**ขั้นตอนที่ 3 — กู้คืน Lost Tracks ด้วย Low-score Detections**

Tracks ที่อยู่ในสถานะ LOST จะถูกนำมาจับคู่กับ Low-score detections อีกครั้ง (IoU ≥ 0.50) ก่อนที่จะถูกลบออก ช่วยให้ระบบติดตาม ID ของบุคคลได้ต่อเนื่องแม้ถูกบดบังชั่วคราว

**ขั้นตอนที่ 4 — อัปเดตสถานะ Track**
- Tracks ที่จับคู่ได้ → อัปเดต bbox, score, hits counter, เพิ่ม center ลง history (สูงสุด 60 จุด)
- Tracks ที่จับคู่ไม่ได้ → เปลี่ยนสถานะเป็น LOST, เพิ่ม time_lost counter
- LOST tracks ที่ time_lost > 30 frames → ลบออกจากระบบ

**ขั้นตอนที่ 5 — สร้าง Track ใหม่**

High-score detections ที่ไม่ถูกจับคู่กับ track ใดเลยจะถูกสร้างเป็น Track ใหม่พร้อม ID ที่ไม่ซ้ำกัน (Singleton counter)

**โครงสร้างข้อมูล Track State Machine**:

```
NEW ──(matched)──► TRACKED ──(unmatched)──► LOST ──(timeout > 30f)──► REMOVED
         ▲                                    │
         └────────────(low-score rescue)───────┘
```

### 2) Bounding Box Ratio Analysis (การตรวจจับท่าทางผิดปกติ — Abnormal Posture)

อัลกอริธึมนี้ตรวจจับสถานะที่บุคคลนอนลงหรือหมดสติ โดยอาศัยหลักการที่ว่า **ร่างกายของคนที่ยืนจะมีสัดส่วนแนวตั้งมากกว่าแนวนอน แต่เมื่อล้มลงสัดส่วนจะกลับกัน**

**ขั้นตอนการคำนวณ**:

1. กรองเฉพาะ keypoints ที่มี visibility > 0.40 และต้องมีอย่างน้อย 6 จุด
2. คำนวณ Bounding Box ครอบทุก keypoint ที่ผ่านการกรอง:
   - bw = max(x) − min(x)
   - bh = max(y) − min(y)
3. คำนวณสัดส่วน:
   $$\text{ratio} = \frac{bw}{bh}$$
4. ตัดสินใจ: ถ้า ratio > 0.72 → abnormal_posture = True

ค่า Threshold 0.72 ถูกเลือกจากการทดสอบเชิงประจักษ์ เพื่อให้ครอบคลุมท่านอน ท่านั่งหมดแรง และท่าล้มทางด้านข้าง

### 3) Keypoint Velocity Analysis (การตรวจจับการล้มกะทันหัน — Sudden Fall)

อัลกอริธึมประเมินความเสี่ยงการล้มจากอัตราการเปลี่ยนแปลงตำแหน่งของ Keypoints ระหว่าง Frame ที่ต่อเนื่องกัน โดยอาศัยหลักการที่ว่าเหตุการณ์ล้มกะทันหันจะก่อให้เกิด displacement ของ Keypoints สูงกว่าปกติอย่างมีนัยสำคัญ

**การคำนวณ**: ระบบเก็บ Keypoint array ขนาด (33, 3) ใน sliding window ขนาด 12 frames จากนั้นคำนวณค่า Mean Euclidean Displacement ของ Keypoints ที่ผ่านเกณฑ์ visibility > 0.5 ในทั้งสอง frame (ต้องมีอย่างน้อย 3 จุด)

$$\bar{v} = \frac{1}{n} \sum_{i=1}^{n} \left\| \mathbf{kp}_i^{(t)} - \mathbf{kp}_i^{(t-1)} \right\|_2, \quad \text{visibility}_i > 0.5$$

โดยพิกัดเป็นค่า normalized (0–1) เทียบกับขนาด frame หาก $\bar{v} > 0.30$ จะตั้งค่า sudden_fall = True

**Temporal Confirmation**: เพื่อลด False Positive จากการเคลื่อนไหวปกติ เช่น การก้มหยิบของ ระบบใช้ Counter สะสมโดยแจ้งเตือนเมื่อตรวจพบ risk ต่อเนื่องครบ 6 frames (FALL_CONFIRM_FRAMES = 6) และลดค่า Counter ลง 1 ทุก frame ที่ไม่พบ risk

### 4) Center of Gravity Trajectory Analysis (การตรวจจับการเดินโซเซ — Gait Anomaly)

อัลกอริธึมนี้ตรวจจับสัญญาณเตือนล่วงหน้า (Early Warning) ก่อนที่บุคคลจะล้ม โดยวิเคราะห์ **ความแปรปรวนของเส้นทางการเคลื่อนที่ของจุดศูนย์ถ่วง (COG)** หลักการคือผู้ที่เริ่มมีภาวะ Heat Stroke หรือเวียนหัวจะเดินโซเซ ทำให้ COG เคลื่อนที่ในเส้นทางที่ผิดปกติ

**ขั้นตอนการคำนวณ**:

1. คำนวณตำแหน่ง COG จาก midpoint ของ LEFT_HIP (23) และ RIGHT_HIP (24):
   $$\text{COG} = \left(\frac{x_{hip\_L} + x_{hip\_R}}{2 \times W},\ \frac{y_{hip\_L} + y_{hip\_R}}{2 \times H}\right)$$
   โดยที่ W, H คือความกว้างและความสูงของ frame (normalized 0–1)

2. เก็บ COG history ไว้ใน deque ขนาด 30 frames (ประมาณ 1 วินาที)

3. คำนวณความแปรปรวนรวม:
   $$\text{variance} = \text{Var}(\text{COG}_x) + \text{Var}(\text{COG}_y)$$
   โดยที่ $\text{Var}(X) = \frac{1}{n}\sum_{i=1}^{n}(x_i - \bar{x})^2$

4. ตัดสินใจ: ถ้า variance > 0.20 → gait_anomaly = True

ระบบจะแสดงเส้น COG Trail ย้อนหลัง 15 จุดบน Frame เพื่อให้ผู้ดูแลเห็น pattern การเคลื่อนไหวแบบ Real-time และแจ้งเตือนระดับเบื้องต้นให้หัวหน้างานตรวจสอบก่อนที่อาการจะรุนแรงถึงขั้นล้มหมดสติ

### 5) Point-in-Polygon Algorithm (การตรวจสอบการบุกรุกพื้นที่)

อัลกอริธึม Point-in-Polygon (PIP) ทำหน้าที่ตรวจสอบความสัมพันธ์เชิงตำแหน่งระหว่างจุด (Point) กับ Polygon แบบ Closed ซึ่งกำหนดเป็นเขตอันตราย (Restricted Zone) โดยระบบ ZENTRA ประยุกต์ใช้ผ่านฟังก์ชัน **cv2.pointPolygonTest()** ของ OpenCV ที่ใช้หลักการ **Ray Casting**

**หลักการ Ray Casting**: อัลกอริธึมยิง Ray ในแนวนอนจากจุดที่ต้องการทดสอบ (cx, cy) ออกไปทางทิศตะวันออกอย่างไม่มีที่สิ้นสุด แล้วนับจำนวนครั้งที่ Ray ตัดผ่านขอบของ Polygon

$$\text{result} =
\begin{cases}
\geq 0 & \text{(ภายใน)} \quad \text{จำนวนครั้งที่ Ray ตัด Polygon เป็นเลขคี่} \\
< 0 & \text{(ภายนอก)} \quad \text{จำนวนครั้งที่ Ray ตัด Polygon เป็นเลขคู่หรือศูนย์}
\end{cases}$$

สำหรับ Polygon ที่มีจุดยอด $n$ จุด $\{P_0, P_1, \ldots, P_{n-1}\}$ การตรวจสอบแต่ละด้าน $\overline{P_i P_{i+1}}$ ว่าตัดกับ Ray หรือไม่ใช้เงื่อนไข:

$$\left(P_i^y \leq c_y < P_{i+1}^y \;\text{ หรือ }\; P_{i+1}^y \leq c_y < P_i^y\right) \;\text{ และ }\; c_x < \frac{(P_{i+1}^x - P_i^x)(c_y - P_i^y)}{P_{i+1}^y - P_i^y} + P_i^x$$

ทำให้อัลกอริธึมมี Time Complexity อยู่ที่ **O(n)** ต่อการตรวจสอบหนึ่งครั้ง ซึ่งเหมาะสมอย่างยิ่งกับการทำงานแบบ Real-time

**การกำหนดและจัดเก็บ Zone**:

ผู้ดูแลระบบกำหนด Restricted Zone บนหน้าจอแบบ Interactive ผ่าน Mouse Callback โดยคลิกซ้ายเพื่อวางจุดยอดของ Polygon (ต้องมีอย่างน้อย 3 จุด รองรับสูงสุด 10 Zones) และคลิกขวาเพื่อยืนยัน Polygon จากนั้นระบบ Serialize พิกัดทุกจุดลงไฟล์ zones.json เพื่อคงสถานะ Zone ไว้ข้ามรอบการทำงาน

**การตรวจสอบใน Pipeline**:

ในแต่ละ Frame ระบบคำนวณ Foot-point ของทุก Active Track จาก ByteTracker โดยใช้จุดศูนย์กลางของ Bounding Box:

$$\mathbf{c}_k = \left(\frac{x_1^{(k)} + x_2^{(k)}}{2},\ \frac{y_1^{(k)} + y_2^{(k)}}{2}\right)$$

จากนั้นประเมิน $\mathbf{c}_k$ กับทุก Zone ที่มีสถานะ ready = True พร้อมกัน หากพบว่า cv2.pointPolygonTest(pts_array, $\mathbf{c}_k$, False) $\geq 0$ สำหรับ Zone ใด ๆ ระบบจะบันทึก Track ID และชื่อ Zone ลงในรายการผู้บุกรุก (Intruder List) และส่งแจ้งเตือนระดับ ALERT พร้อม Timestamp และภาพ Frame ขณะเกิดเหตุผ่าน LINE OA ทันที

## 7.2.3 สถาปัตยกรรมซอฟต์แวร์ (Software Architecture)

### 1) ระบบประมวลผลแบบขนาน (Multi-threaded Asynchronous Pipeline)

โดยปกติหากระบบทำงานตามลำดับขั้นตอน เช่น รอรับภาพ → รอ AI วิเคราะห์ → รอส่ง LINE จะทำให้การแสดงผลบนหน้าจอสะดุด เนื่องจากการส่งภาพไปให้ AI วิเคราะห์และการส่ง LINE ใช้เวลานานกว่าการแสดงผลมาก ระบบ ZENTRA จึงแบ่งการทำงานออกเป็น 3 ส่วนที่ทำงานพร้อมกันอย่างอิสระ (Multi-threaded) โดยแต่ละส่วนรับ-ส่งข้อมูลกันผ่านคิว (Queue) ซึ่งทำหน้าที่เป็นสายพานลำเลียงข้อมูลระหว่างกัน

```
กล้อง / RTSP
     │
     ▼
┌─────────────────┐  Queue(4)   ┌──────────────────┐  Queue(4)
│  FrameReader    ├────────────►│ InferenceWorker   ├──────────►┐
│  (อ่านภาพ)     │             │ (วิเคราะห์ AI)    │           │
└─────────────────┘             └──────────────────┘           │
                                                                ▼
                                                     Main Loop (แสดงผล+ตรวจสอบ Zone)
                                                                │
                                                                ▼
                                                     ┌──────────────────┐
                                                     │  AlertSender     │──► LINE OA
                                                     │  (ส่งแจ้งเตือน) │
                                                     └──────────────────┘
```

1. **FrameReader (ตัวอ่านภาพ)** — ทำหน้าที่อ่านภาพจากกล้องตลอดเวลาและเก็บรอไว้ในคิว (Queue — แถวรอข้อมูล) สูงสุด 4 ภาพ หากคิวเต็ม ระบบจะลบภาพเก่าที่สุดทิ้งและนำภาพใหม่เข้าแทน เพื่อให้การแสดงผลใช้ภาพล่าสุดเสมอ ไม่ค้างอยู่กับภาพที่ล่าช้า

2. **InferenceWorker (ตัววิเคราะห์ AI)** — ทำหน้าที่รับภาพแล้วส่งให้โมเดล AI วิเคราะห์ทุก 3 ภาพ เพื่อลดภาระการประมวลผล (CPU — หน่วยประมวลผลกลางของคอมพิวเตอร์) โดยส่งพร้อมกัน 2 โมเดล คือ ตรวจ PPE และตรวจการล้ม จากนั้นส่งผลลัพธ์กลับมาให้ส่วนแสดงผล การทำงานแบบนี้ทำให้การรอ AI ไม่ส่งผลให้ภาพบนหน้าจอสะดุด

3. **AlertSender (ตัวส่งแจ้งเตือน)** — ทำหน้าที่รับคำสั่งแจ้งเตือนจากคิวแจ้งเตือน (Alert Queue) แล้วดำเนินการส่ง LINE OA โดยแนบภาพหลักฐานขณะเกิดเหตุ หากส่งไม่สำเร็จระบบจะลองส่งซ้ำสูงสุด 3 ครั้ง (รอ 2 วินาทีต่อครั้ง) การแยกส่วนนี้ออกมาทำให้การรอสัญญาณอินเทอร์เน็ตไม่กระทบต่อการแสดงผลบนหน้าจอ

### 2) ระบบจัดลำดับความสำคัญและการหน่วงเวลาแจ้งเตือน (Alert Priority & Cooldown System)

ในสภาพแวดล้อมโรงงานที่มีการเคลื่อนไหวของบุคคลหลายคนพร้อมกัน หากระบบส่งการแจ้งเตือนทุกครั้งที่ตรวจพบเหตุการณ์โดยไม่มีการกรอง จะเกิดการแจ้งเตือนซ้ำซ้อนในปริมาณมากจนผู้รับไม่สามารถตอบสนองได้ทัน ระบบ ZENTRA จึงออกแบบกลไก 2 ส่วนเพื่อแก้ปัญหานี้

**ส่วนที่ 1 — การจัดลำดับความรุนแรง (Priority Level)**

การแจ้งเตือนถูกแบ่งออกเป็น 3 ระดับตามความเร่งด่วนของเหตุการณ์ และแต่ละระดับจะส่งถึงกลุ่มผู้รับที่เกี่ยวข้องโดยตรงเท่านั้น เพื่อให้ผู้รับสามารถตัดสินใจได้อย่างรวดเร็วโดยไม่ถูกรบกวนจากการแจ้งเตือนที่ไม่อยู่ในความรับผิดชอบของตน

| ระดับ | เงื่อนไขการเกิดเหตุ | ระยะหน่วงเวลา | กลุ่มผู้รับ |
|-------|-------------------|--------------|------------|
| WARNING (เฝ้าระวัง) | ตรวจพบการไม่สวม PPE | 30 วินาที | หัวหน้างาน |
| ALERT (ตักเตือน) | ตรวจพบการเข้าพื้นที่อันตราย | 20 วินาที | เจ้าหน้าที่ความปลอดภัย + หัวหน้างาน |
| EMERGENCY (ฉุกเฉิน) | ตรวจพบการล้มหรือหมดสติ | 15 วินาที | ทีมฉุกเฉิน + เจ้าหน้าที่ความปลอดภัย + หัวหน้างาน |

**ส่วนที่ 2 — การหน่วงเวลาแจ้งเตือน (Cooldown)**

Cooldown หรือ ระยะเวลาหน่วง คือช่วงเวลาที่ระบบจะไม่ส่งการแจ้งเตือนซ้ำในประเภทเดียวกัน นับจากครั้งล่าสุดที่ส่งไป กลไกนี้ทำงานโดยบันทึกเวลา Unix Timestamp ของการแจ้งเตือนล่าสุดในแต่ละประเภท แล้วเปรียบเทียบกับเวลาปัจจุบันทุกครั้งก่อนส่ง

$$\text{อนุญาตให้ส่ง} = \left(t_{\text{ปัจจุบัน}} - t_{\text{ล่าสุด}}\right) \geq T_{\text{cooldown}}$$

ตัวอย่างเช่น หากกล้องจับภาพคนงานที่ไม่สวมหมวกในโซนเดิมตลอด 30 วินาที ระบบจะส่งการแจ้งเตือนเพียงครั้งเดียว ไม่ใช่ทุก Frame ซึ่งอาจทำให้ Line Group ของหัวหน้างานถูกส่งข้อความนับร้อยครั้งภายในเวลาไม่กี่วินาที โดยระดับ EMERGENCY ตั้งค่า Cooldown ไว้สั้นที่สุด (15 วินาที) เพื่อให้ยังสามารถแจ้งเตือนซ้ำได้หากเหตุการณ์ยังดำเนินอยู่และยังไม่มีผู้เข้าช่วยเหลือ

### 3) ระบบเก็บข้อมูลและสร้างคำอธิบายภาพอัตโนมัติ (Auto Data Collection & YOLO Annotation System)

โมเดล AI ที่ฝึกมาจาก Dataset ทั่วไปอาจมีความแม่นยำลดลงเมื่อนำไปใช้งานในสภาพแสง มุมกล้อง และชุดแรงงานที่แตกต่างกันในแต่ละโรงงาน ระบบ ZENTRA จึงมีกลไกเก็บภาพจากกล้องโรงงานจริงพร้อมสร้างข้อมูลคำอธิบาย (Annotation หรือ ป้ายกำกับตำแหน่งวัตถุ) โดยอัตโนมัติทุกครั้งที่เกิดเหตุการณ์ เพื่อนำไปปรับแต่งโมเดลให้แม่นยำยิ่งขึ้นในภายหลัง

**หมวดหมู่การเก็บข้อมูล**

ระบบแบ่งการจัดเก็บออกเป็น 4 หมวดหมู่ตามประเภทเหตุการณ์ที่เกิดขึ้น

| หมวดหมู่ | เงื่อนไขการบันทึก | หมายเหตุ |
|---------|-----------------|---------|
| ppe_violations/ | ตรวจพบการไม่สวม PPE | บันทึกเฉพาะ Frame ที่มี violation |
| zone_intrusions/ | ตรวจพบการบุกรุกพื้นที่อันตราย | บันทึกเฉพาะ Frame ที่มีการละเมิด |
| fall_events/ | ตรวจพบการล้มหรือหมดสติ | บันทึกทันทีโดยไม่มีเงื่อนไข (Force Mode) |
| normal/ | ไม่มีเหตุการณ์ผิดปกติ | บันทึกทุก ๆ 300 Frame เพื่อให้ Dataset มีความสมดุล |

การเก็บ Frame ปกติในหมวด normal/ มีความสำคัญเพื่อป้องกันปัญหา Class Imbalance หรือ ความไม่สมดุลของข้อมูล กล่าวคือหากชุดข้อมูลมีแต่ภาพที่มีความผิดปกติ โมเดลอาจเรียนรู้ไม่ได้ว่าสภาวะปกติมีลักษณะอย่างไร

**รูปแบบคำอธิบายภาพแบบ YOLO**

ทุกครั้งที่บันทึกภาพ ระบบจะสร้างไฟล์คำอธิบาย (Annotation File นามสกุล .txt) คู่กันโดยอัตโนมัติ โดยใช้รูปแบบมาตรฐาน YOLO ซึ่งบันทึกตำแหน่งและขนาดของวัตถุในภาพเป็นค่าสัดส่วน (Normalized Coordinates) ระหว่าง 0.0 ถึง 1.0 เทียบกับขนาดภาพจริง ทำให้ข้อมูลใช้ได้กับภาพที่มีความละเอียดต่างกัน

```
<รหัสประเภทวัตถุ> <ตำแหน่งกึ่งกลาง X> <ตำแหน่งกึ่งกลาง Y> <ความกว้าง> <ความสูง>
```

ตัวอย่างเช่น หากตรวจพบ "ไม่สวมหมวก" ที่กึ่งกลางภาพ ขนาดกรอบ 20% ของความกว้าง และ 30% ของความสูง จะได้ไฟล์ .txt ดังนี้

```
7 0.500 0.400 0.200 0.300
```

**ระบบจำกัดโควตาและความปลอดภัยของข้อมูล (Quota & Thread-safety)**

เพื่อป้องกันพื้นที่จัดเก็บถูกใช้งานจนเต็ม ระบบกำหนดโควตาสูงสุด 2,000 ภาพต่อหมวดหมู่ โดยนับจำนวนภาพที่มีอยู่จริงบน Disk ทุกครั้งที่เปิดระบบ และใช้กลไก Threading Lock หรือ ตัวล็อกสำหรับการทำงานหลายเธรด เพื่อป้องกันการนับโควตาผิดพลาดเมื่อหลาย Thread พยายามบันทึกภาพพร้อมกัน

เมื่อเก็บข้อมูลได้เพียงพอ ระบบรองรับการส่งออกเป็นรูปแบบ COCO JSON ซึ่งเป็นมาตรฐานที่แพลตฟอร์ม Roboflow รองรับ เพื่อนำไปฝึกโมเดลรอบใหม่ด้วยข้อมูลจากโรงงานจริง
